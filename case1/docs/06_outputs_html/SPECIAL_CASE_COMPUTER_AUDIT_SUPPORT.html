Special Case Computer Audit Support: 疑似違建 / 未合法申報施工

文件狀態：內部工作稿 定位：Computer Audit 稽核支援設計 適用案例：工程承攬專案稽核 Special Case 重要聲明：本文件用於規劃電腦稽核如何協助整理資料、比對時序、產出例外清單與證據鏈，不做法律判定，也不取代稽核主管、法務、建築師、專業技師或主管機關之判斷。

1. 核心定位

本文件說明 Computer Audit 可以如何協助工程承攬專案稽核，特別是「疑似違建 / 未合法申報施工」這類跨文件、跨系統、跨現場的治理風險。

Computer Audit 的角色：

  • 整理資料來源與欄位。
  • 比對文件、ERP、Workflow 與現場紀錄。
  • 建立 Suspicious Timeline。
  • 產出例外清單與 Red Flags。
  • 整理專案證據鏈與待補資料。

Computer Audit 不應做：

  • 自動判定違建。
  • 自動判定法律責任。
  • 自動否決估驗或請款。
  • 自動替代稽核結論。
  • 自動對外通報或執法。

2. 為什麼需要 Computer Audit

這類專案稽核的難點，不在於單一資料難懂，而在於資料分散：

  • 合約在 DMS。
  • 圖說在圖管系統或附件。
  • 施工日報在現場系統。
  • 估驗在 ERP。
  • 請款在 AR。
  • 簽核在 Workflow。
  • 現場照片可能在手機、雲端或附件中。
  • 法務判斷可能在 Email 或會議紀錄中。

Computer Audit 的價值，是把這些資料整理成稽核人員看得懂、可追溯、可覆核的證據鏈視圖。

3. 稽核支援工作

3.1 文件比對

可協助比對：

  • 合約範圍與變更單內容。
  • 合約附件圖說與核准圖說版本。
  • 估驗單項目與合約 / BOQ 項目。
  • 驗收文件與請款、保留款釋放條件。
  • 文件中的日期、金額、地點、專案代號是否一致。

輸出應為：

  • 不一致清單。
  • 需人工確認項目。
  • 缺失或缺件提示。
  • 文件版本對照表草案。

不應輸出：

  • 「本案確定違建」。
  • 「本案合法」。
  • 「可直接付款」。

3.2 圖說與現場證據索引

可協助：

  • 將圖說版本、圖名、日期、編號整理成索引。
  • 將現場照片依日期、位置、施工項目建立索引。
  • 標示可能不一致的施工範圍、位置或項目。
  • 整理需要建築師、監造或專業技師確認的疑點。

限制：

  • 圖說、比例尺、施工照片角度與現場狀態仍需專業人員確認。
  • 輸出只能作為「疑點整理」，不能作為工程專業結論。

3.3 Suspicious Timeline

Suspicious Timeline 是本案例的核心工具，用來檢查是否存在「先做、後補、時序異常」。

輸入資料：

  • 合約簽核日期。
  • 建照或合法性文件日期。
  • 圖說版本日期。
  • 變更單建立與核准日期。
  • 施工日報日期。
  • 估驗申請與核准日期。
  • 請款與發票日期。
  • 收款日期。
  • 驗收日期。
  • 保留款釋放日期。

檢查重點：

  • 是否先施工後變更。
  • 是否先估驗後核准。
  • 是否先請款後估驗。
  • 是否先釋放保留款後驗收。
  • 是否附件補傳日期晚於簽核完成日期。

3.4 Workflow 分析

可協助：

  • 整理 Workflow 節點與實際簽核順序。
  • 找出退回、補件、跳關、補簽核紀錄。
  • 比對簽核者角色是否符合 RACI。
  • 標示附件補傳與核准時間差。
  • 檢查例外放行是否有理由與授權。

不應：

  • 自動認定簽核有效或無效。
  • 自動判斷某主管是否失職。

3.5 ERP 交叉分析

可協助建立專案資料鏈：

Project -> Contract -> Change Order -> Valuation -> Billing -> AR -> Collection -> Retention

交叉檢查重點：

  • 合約金額、變更金額、估驗金額、請款金額與收款金額是否一致。
  • 是否有未核准變更卻已估驗的項目。
  • 是否有合法性文件欄位空白但已請款的專案。
  • 是否有估驗、請款、收款時序不合理的專案。

4. Suspicious Timeline 欄位

欄位來源系統稽核用途
合約簽核日期Workflow / DMS確認專案授權起點
合法性文件日期DMS / 專案主檔比對文件日期與施工日期
圖說版本日期DMS / 圖管系統建立圖說版本索引
變更單核准日期Workflow / ERP比對施工與估驗時間
施工日報日期現場系統抽取施工項目與位置
估驗日期ERP比對合約、變更與施工證據
請款日期ERP / AR比對估驗核准日期
收款日期ERP / 銀行對帳比對請款與沖帳
驗收日期Workflow / DMS比對保留款釋放
保留款釋放日期ERP檢查是否符合驗收條件

5. 例外規則初版

規則例外提示
施工日早於合法性文件核准日可能存在未合法申報即施工，需人工確認
施工日早於變更單核准日可能存在先施工後補變更
估驗日早於變更單核准日未核准變更可能已被計價
請款日早於估驗核准日請款基礎可能不足
保留款釋放日早於驗收日保留款釋放條件可能未成立
附件上傳日晚於簽核完成日可能存在後補附件

6. 建議輸出

輸出用途
Evidence Map將文件、ERP、Workflow、現場資料串成證據地圖
Suspicious Timeline顯示先做後補、時序異常與補件風險
Red Flag List列出需要人工查核的疑點
Missing Evidence List列出缺少的文件、附件、欄位或 Log
Cross-system Consistency Report比對 ERP、Workflow、DMS 與現場資料
Project Risk Summary給稽核主管的風險摘要
Manual Review Queue將無法由資料直接判斷的事項排入人工覆核

7. 人工覆核規則

Computer Audit 產出後，必須由稽核主管或稽核人員執行：

  • 查核來源資料是否完整。
  • 判斷例外是否成立。
  • 訪談相關權責單位。
  • 必要時請法務、建築師、專業技師或主管機關協助確認。
  • 形成稽核發現、待釐清事項或改善建議。

稽核報告需明確標示：

  • 資料來源。
  • 檢查規則。
  • 系統限制。
  • 不確定事項。
  • 人工覆核結論。

8. 使用風險與控制

風險說明控制方式
資料不完整文件缺漏可能導致錯誤判斷缺件需列為 Missing Evidence
圖說判讀限制圖面與照片需專業判讀圖說與現場判讀需專業人員確認
法律過度推論疑點不等於法律結論禁止輸出法律判定語句
權限與隱私合約、照片、個資、工程資料可能敏感需控管資料存取與遮罩
自動化過度例外清單被誤用為自動決策設定人工覆核關卡

9. 成功判準

本測試案例成功，不是因為系統判定了違建，而是因為：

  • 能協助找出文件缺口。
  • 能整理 Suspicious Timeline。
  • 能比對 ERP / Workflow / DMS / 現場資料。
  • 能讓稽核主管更快形成查核方向。
  • 能保留清楚的證據鏈與人工判斷紀錄。
Computer Audit 的價值，在於幫助專案稽核看見「系統正常表象下的治理斷點」，而不是替人做法律判斷。
