# 工程承攬業銷售與收款循環治理作業辦法 Computer Audit 附加版本 v0.2

- 文件狀態：個人工作稿 / v0.2 Draft / 稽核主管校閱版
- 文件定位：Computer Audit 附加深度版
- 對應主文件：傳統治理主文件
- 主要讀者：公司稽核主管、電腦稽核、內部稽核、資訊、ERP Owner、Workflow Owner
- 文件邊界：不是另一套制度，不是對外服務提案，不是全自動稽核規格

## 0. 文件聲明

本文件是《工程承攬業銷售與收款循環治理作業辦法 v0.1》的 Computer Audit 附加深度版。

本文件不是另一套制度，而是針對主文件所定義的同一治理主題，補充以下內容：

- ERP / Workflow 控制。
- Audit Trail。
- Log。
- SoD。
- Computer Audit 測試規則。
- Suspicious Timeline。
- 異常 Red Flags。
- 數位證據鏈。

本文件不進行法律判定，不建立全自動稽核，也不取代稽核主管與稽核人員的專業判斷。

## 2. 漸進導入原則

電腦稽核導入必須循序漸進。

```text
內控 / SOP
  -> Computer Audit
  -> Workflow / ERP
  -> 系統留痕與例外清單
  -> 管理追蹤與稽核覆核
```

本文件目前只處理：

- Computer Audit。
- Workflow / ERP 可稽核性。
- 系統留痕與資料追溯。
- 稽核例外清單與證據鏈整理。

本文件不直接導入：

- 全自動稽核。
- 系統自動認定。
- 系統自動核准。
- 對外展示包裝。

## 3. Computer Audit 設計原則

本文件的 Computer Audit 不是為了炫技，而是為了支撐治理。

設計原則：

- 先對應主文件 KCP，再設計資料測試。
- 先檢查流程是否一致，再檢查技術細節。
- 先做例外清單，再做原因分析。
- 先讓稽核主管與稽核人員看得懂，再談自動化。
- 每一個測試點都要能回扣流程、權責、文件或表單。
- CA 結果只能作為異常提示，不直接等同缺失結論。

## 4. 核心資料鏈

工程承攬業銷售與收款循環的基本資料鏈如下：

```text
Project_ID
  -> Contract_ID
  -> Change_Order_ID
  -> Valuation_ID
  -> Billing_ID
  -> Invoice_ID
  -> AR_ID
  -> Collection_ID
  -> Retention_ID
  -> Acceptance_ID
  -> Exception_ID
```

Computer Audit 的第一個任務，是確認這些資料鏈是否能串起來。

常見問題：

- 系統有專案，但缺合約附件。
- 有估驗，但缺變更核准。
- 有請款，但缺估驗依據。
- 有收款，但未沖帳或沖錯專案。
- 有保留款釋放，但缺驗收依據。
- 有 Workflow 簽核，但附件是事後補上。

## 5. Audit Trail 黃金欄位

以下欄位是後續建立系統留痕與電腦稽核追溯能力的最低基礎。

| 欄位 | 說明 | 治理目的 |
| --- | --- | --- |
| Record_ID | 系統交易或主檔唯一識別碼 | 追溯單筆紀錄 |
| Project_ID | 專案代號 | 串聯專案生命週期 |
| Contract_ID | 合約代號 | 串聯合約與收入基礎 |
| Source_Document_ID | 原始文件代號 | 串聯文件證據 |
| Workflow_Ref_ID | Workflow 簽核代號 | 確認授權來源 |
| Created_By | 建立人 | 確認責任歸屬 |
| Created_At | 建立時間 | 檢查時序 |
| Approved_By | 核准人 | 檢查授權 |
| Approved_At | 核准時間 | 檢查先後順序 |
| Changed_By | 異動人 | 檢查敏感異動 |
| Changed_At | 異動時間 | 檢查補登或後改 |
| Change_Reason | 異動原因 | 檢查合理性 |
| Attachment_Hash | 附件雜湊或版本識別 | 防止附件被替換 |
| Status_Code | 狀態碼 | 對齊 ERP / Workflow 狀態 |

## 6. SoD 檢查原則

### 6.1 不相容職責

| 不相容組合 | 風險 | 檢查方向 |
| --- | --- | --- |
| 合約建立 + 合約核准 | 未授權合約可能被自行建立 | 比對建立人與核准人 |
| 估驗提出 + 估驗核准 | 進度可能未經覆核 | 比對 PM、工程主管與核准人 |
| 請款提出 + 發票開立 + AR 建立 | 無基礎請款可能進入財務紀錄 | 檢查角色權限與交易人 |
| 收款確認 + 沖帳 | 自行消滅應收或錯沖 | 檢查財會角色分工 |
| 保留款釋放提出 + 核准 | 未達條件釋放 | 檢查驗收與核准人 |
| ERP 主檔維護 + 業務核准 | 主檔與授權可能混淆 | 檢查資訊與業務權限 |

### 6.2 權限治理

最低要求：

- 高風險權限應定期複核。
- 離職、調職、代理權限應有啟用與停用紀錄。
- 特權帳號異動應留下原因與核准紀錄。
- 同一人若具有不相容權限，應列入例外清單並由主管核准。

## 8. Suspicious Timeline

Suspicious Timeline 是用來檢查「先做、後補、時序異常」的治理工具。

### 8.1 核心日期

| 日期 | 來源 | 用途 |
| --- | --- | --- |
| 合約核准日 | Workflow / DMS | 確認專案授權起點 |
| ERP 專案建立日 | ERP Project | 檢查是否先建案 |
| 施工開始日 | 施工日報 / PM 紀錄 | 檢查是否先施工 |
| 變更單建立日 | ERP / Workflow | 確認變更是否被正式提出 |
| 變更單核准日 | Workflow | 確認變更是否先核准 |
| 估驗申請日 | ERP Valuation | 檢查估驗是否早於核准 |
| 估驗核准日 | Workflow | 確認覆核完成 |
| 請款申請日 | ERP Billing | 檢查請款時點 |
| 發票開立日 | Invoice | 檢查是否早於核准請款 |
| 收款日 | Bank / Collection | 檢查沖帳與應收 |
| 驗收日 | Acceptance | 檢查保留款釋放條件 |
| 附件上傳日 | DMS | 檢查是否後補附件 |

### 8.2 可疑時序規則

| 規則 | 可能風險 |
| --- | --- |
| ERP 專案建立日早於合約核准日 | 先建案後補合約 |
| 施工開始日早於合約核准日 | 未授權施工 |
| 估驗申請日早於變更核准日 | 先施工後補變更 |
| 請款申請日早於估驗核准日 | 未核准即請款 |
| 發票開立日早於請款核准日 | 先開票後補流程 |
| 保留款釋放日早於驗收日 | 未達條件釋放 |
| 附件上傳日晚於 Workflow 核准日 | 可能後補附件 |

### 8.3 稽核使用方式

Suspicious Timeline 不直接判定缺失。

它的用途是：

- 產出異常清單。
- 協助稽核人員選樣。
- 提醒需補查的文件。
- 建立專案生命週期證據鏈。
- 支援稽核人員判斷。

## 9. Red Flags（CA 深度版）

| Red Flag | 技術觀察 | 治理意義 |
| --- | --- | --- |
| ERP 有專案但缺合約 Workflow | 系統主檔與授權斷裂 | 專案可能未正式核准 |
| 合約金額被修改但缺異動原因 | Audit Log 無 Reason Code | 主檔異動治理不足 |
| 估驗缺 Workflow_Ref_ID | 孤兒交易 | 估驗未經正式核准 |
| 請款對應不到 Valuation_ID | 資料鏈斷裂 | 可能無基礎請款 |
| 發票早於請款核准 | 時序異常 | 先執行後補流程 |
| 收款長期未沖帳 | Aging 異常 | AR 管理或沖帳流程失效 |
| 特權帳號修改高風險欄位 | Admin Log 異常 | 後台操作風險 |
| 附件上傳晚於核准時間 | DMS 時戳異常 | 可能後補附件 |
| 同一人具請款與沖帳權限 | RBAC 異常 | SoD 失效 |
| 爭議款仍列正常 AR | AR 狀態碼異常 | 財報與管理分類風險 |

## 10. 稽核輔助整理範圍

本文件可搭配一般資料整理工具、試算表、ERP 報表與文件比對工具使用，但任何工具都不得自動作成稽核結論。

工具可協助：

- 文件比對。
- 表單完整性檢查。
- 時序排序。
- Workflow 狀態摘要。
- ERP 例外清單整理。
- Audit Trail 摘要。
- Red Flags 分類。
- 證據鏈整理。
- 專案風險摘要草稿。

工具不可負責：

- 判定缺失成立。
- 判定違法或違建。
- 核准交易。
- 取代主管覆核。
- 取代內部稽核專業判斷。
- 自動執行稽核處置。

最終判斷仍由稽核主管與稽核人員負責。

## 11. 證據鏈整理

### 11.1 基本證據鏈

每一項 CA 異常至少應整理：

- Project_ID。
- 交易或文件編號。
- 異常類型。
- 對應 KCP。
- 對應主文件章節。
- 來源系統。
- 來源表單或附件。
- 相關日期。
- 相關人員。
- 建議補查文件。
- 稽核人員結論。

### 11.2 證據鏈呈現格式

| 欄位 | 說明 |
| --- | --- |
| Finding_Candidate_ID | CA 例外清單編號 |
| Related_KCP | 對應主文件 KCP |
| Exception_Type | 異常類型 |
| Data_Source | ERP / Workflow / DMS / Bank / Manual |
| Evidence_Link | 文件或系統連結 |
| Timeline_Summary | 時序摘要 |
| Risk_Description | 風險說明 |
| Auditor_Action | 稽核建議動作 |
| Auditor_Conclusion | 稽核人員判斷結論 |

## 12. 導入優先順序

建議先做小範圍可落地測試，不一次展開全循環。

| 優先順序 | 模組 | 理由 |
| --- | --- | --- |
| 1 | SC-02 工程估驗計價與營收認列 | 財報重大性高，資料鏈清楚 |
| 2 | SC-04 請款與發票管理 | 可直接比對估驗、請款、發票與 AR |
| 3 | SC-06 應收帳款與保留款管理 | 工程承攬業特色明顯 |
| 4 | SC-03 契約變更 | 先做後補風險高 |
| 5 | SC-08 異常、逾期與爭議 | 適合建立管理監督與缺失追蹤 |

## 13. 稽核主管校閱待確認事項

本文件尚待公司稽核主管與相關權責單位校閱。至少需確認：

- 現行 ERP 是否具備必要 Project_ID / Contract_ID / Workflow_Ref_ID。
- Workflow 是否可提供核准日期、核准人、退回與補件紀錄。
- DMS 是否可提供附件版本、上傳日與附件識別。
- RBAC 是否可匯出角色、權限與使用者清單。
- 高風險欄位是否已有異動 Log。
- SoD 衝突清單是否能由資訊單位定期提供。
- CA 例外清單應由誰覆核與結案。
- 例外清單應如何與現行內稽底稿銜接。

## 14. 結語

本文件的價值，是讓主文件中的治理要求可以被資料、系統與稽核測試支持。

主文件回答：「企業應如何治理？」

CA 附加版回答：「系統與資料能否證明治理真的被執行？」

下一階段應先完成 Computer Audit 的最小可行測試，再決定是否納入正式內稽作業或例外追蹤機制。
