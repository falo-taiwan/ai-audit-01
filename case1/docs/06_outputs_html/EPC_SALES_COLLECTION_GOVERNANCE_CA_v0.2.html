CA 版先看這四件事

資料能不能串起來？

專案、合約、變更、估驗、請款、收款、保留款與驗收是否可追溯。

流程有沒有真的跑？

Workflow、附件、核准時間與系統狀態是否支持主文件要求。

哪些例外要提示？

先建案後核准、先開票後補簽、無附件估驗、未驗收釋放保留款。

誰做最後判斷？

系統只產生例外與證據鏈，最後仍由稽核主管與稽核人員判斷。

0. 文件聲明

本文件是《工程承攬業銷售與收款循環治理作業辦法 v0.1》的 Computer Audit 附加深度版。

本文件不是另一套制度，而是針對主文件所定義的同一治理主題，補充以下內容：

  • ERP / Workflow 控制。
  • Audit Trail。
  • Log。
  • SoD。
  • Computer Audit 測試規則。
  • Suspicious Timeline。
  • 異常 Red Flags。
  • 數位證據鏈。

本文件不進行法律判定，不建立全自動稽核，也不取代稽核主管與稽核人員的專業判斷。

2. 漸進導入原則

電腦稽核導入必須循序漸進。

內控 / SOP
  -> Computer Audit
  -> Workflow / ERP
  -> 系統留痕與例外清單
  -> 管理追蹤與稽核覆核

本文件目前只處理：

  • Computer Audit。
  • Workflow / ERP 可稽核性。
  • 系統留痕與資料追溯。
  • 稽核例外清單與證據鏈整理。

本文件不直接導入：

  • 全自動稽核。
  • 系統自動認定。
  • 系統自動核准。
  • 對外展示包裝。

3. Computer Audit 設計原則

本文件的 Computer Audit 不是為了炫技，而是為了支撐治理。

設計原則：

  • 先對應主文件 KCP，再設計資料測試。
  • 先檢查流程是否一致，再檢查技術細節。
  • 先做例外清單，再做原因分析。
  • 先讓稽核主管與稽核人員看得懂，再談自動化。
  • 每一個測試點都要能回扣流程、權責、文件或表單。
  • CA 結果只能作為異常提示，不直接等同缺失結論。

4. 核心資料鏈

工程承攬業銷售與收款循環的基本資料鏈如下：

Project_ID
  -> Contract_ID
  -> Change_Order_ID
  -> Valuation_ID
  -> Billing_ID
  -> Invoice_ID
  -> AR_ID
  -> Collection_ID
  -> Retention_ID
  -> Acceptance_ID
  -> Exception_ID

Computer Audit 的第一個任務，是確認這些資料鏈是否能串起來。

常見問題：

  • 系統有專案，但缺合約附件。
  • 有估驗，但缺變更核准。
  • 有請款，但缺估驗依據。
  • 有收款，但未沖帳或沖錯專案。
  • 有保留款釋放，但缺驗收依據。
  • 有 Workflow 簽核，但附件是事後補上。

6. SoD 檢查原則

6.1 不相容職責

不相容組合風險檢查方向
合約建立 + 合約核准未授權合約可能被自行建立比對建立人與核准人
估驗提出 + 估驗核准進度可能未經覆核比對 PM、工程主管與核准人
請款提出 + 發票開立 + AR 建立無基礎請款可能進入財務紀錄檢查角色權限與交易人
收款確認 + 沖帳自行消滅應收或錯沖檢查財會角色分工
保留款釋放提出 + 核准未達條件釋放檢查驗收與核准人
ERP 主檔維護 + 業務核准主檔與授權可能混淆檢查資訊與業務權限

6.2 權限治理

最低要求：

  • 高風險權限應定期複核。
  • 離職、調職、代理權限應有啟用與停用紀錄。
  • 特權帳號異動應留下原因與核准紀錄。
  • 同一人若具有不相容權限，應列入例外清單並由主管核准。

8. Suspicious Timeline

Suspicious Timeline 是用來檢查「先做、後補、時序異常」的治理工具。

8.1 核心日期

日期來源用途
合約核准日Workflow / DMS確認專案授權起點
ERP 專案建立日ERP Project檢查是否先建案
施工開始日施工日報 / PM 紀錄檢查是否先施工
變更單建立日ERP / Workflow確認變更是否被正式提出
變更單核准日Workflow確認變更是否先核准
估驗申請日ERP Valuation檢查估驗是否早於核准
估驗核准日Workflow確認覆核完成
請款申請日ERP Billing檢查請款時點
發票開立日Invoice檢查是否早於核准請款
收款日Bank / Collection檢查沖帳與應收
驗收日Acceptance檢查保留款釋放條件
附件上傳日DMS檢查是否後補附件

8.2 可疑時序規則

規則可能風險
ERP 專案建立日早於合約核准日先建案後補合約
施工開始日早於合約核准日未授權施工
估驗申請日早於變更核准日先施工後補變更
請款申請日早於估驗核准日未核准即請款
發票開立日早於請款核准日先開票後補流程
保留款釋放日早於驗收日未達條件釋放
附件上傳日晚於 Workflow 核准日可能後補附件

8.3 稽核使用方式

Suspicious Timeline 不直接判定缺失。

它的用途是：

  • 產出異常清單。
  • 協助稽核人員選樣。
  • 提醒需補查的文件。
  • 建立專案生命週期證據鏈。
  • 支援稽核人員判斷。

10. 稽核輔助整理範圍

本文件可搭配一般資料整理工具、試算表、ERP 報表與文件比對工具使用，但任何工具都不得自動作成稽核結論。

工具可協助：

  • 文件比對。
  • 表單完整性檢查。
  • 時序排序。
  • Workflow 狀態摘要。
  • ERP 例外清單整理。
  • Audit Trail 摘要。
  • Red Flags 分類。
  • 證據鏈整理。
  • 專案風險摘要草稿。

工具不可負責：

  • 判定缺失成立。
  • 判定違法或違建。
  • 核准交易。
  • 取代主管覆核。
  • 取代內部稽核專業判斷。
  • 自動執行稽核處置。

最終判斷仍由稽核主管與稽核人員負責。

12. 導入優先順序

建議先做小範圍可落地測試，不一次展開全循環。

優先順序模組理由

13. 稽核主管校閱待確認事項

本文件尚待公司稽核主管與相關權責單位校閱。至少需確認：

  • 現行 ERP 是否具備必要 Project_ID / Contract_ID / Workflow_Ref_ID。
  • Workflow 是否可提供核准日期、核准人、退回與補件紀錄。
  • DMS 是否可提供附件版本、上傳日與附件識別。
  • RBAC 是否可匯出角色、權限與使用者清單。
  • 高風險欄位是否已有異動 Log。
  • SoD 衝突清單是否能由資訊單位定期提供。
  • CA 例外清單應由誰覆核與結案。
  • 例外清單應如何與現行內稽底稿銜接。
