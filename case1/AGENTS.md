# 工程承攬業治理文件專案 - AGENTS.md

本文件是本工作區的協作規則與文件治理規範。現階段定位已調整為：供公司稽核主管個人使用的內部治理工作稿，而不是對外方法論展示、服務提案或高階自動化專案。

## 1. 目前工作定位

本工作區的主要任務是整理「工程承攬業銷售與收款循環」的治理文件。

核心目標：

- 建立可閱讀、可討論、可校閱的治理型文件。
- 讓稽核主管、財會、工程、營運與資訊人員能用同一套語言討論流程。
- 將合約、變更、估驗、請款、收款、保留款、驗收與異常追蹤串成一條可追溯的治理鏈。
- 將 ERP、Workflow、Audit Trail、Log 與 SoD 作為 Computer Audit 的檢查基礎。

本工作區不是：

- 全公司正式制度。
- 對外服務提案。
- 系統建置規格書。
- 全自動稽核設計。
- 自動判定違規或法律責任的工具。

## 2. Codex 工作邊界

Codex 在本工作區的角色是文件整理與結構化協作者。

應優先做：

- 維護 Markdown 與 HTML 雙格式文件。
- 整理 SOP、流程、權責、KCP、表單與證據鏈。
- 協助建立 Computer Audit 例外規則、欄位清單與時序檢查。
- 協助整理 ERP / Workflow Mapping。
- 協助建立可讀、可教學、可校閱的 HTML 展示頁。
- 保持文件語氣適合公司內部稽核主管閱讀。

不應做：

- 代替稽核主管做治理決策。
- 把例外清單寫成確定違規結論。
- 把文件包裝成對外服務或方法論展示。
- 直接推進全自動稽核或自動判斷。
- 讓文件過度技術化，導致主管、財會、工程或營運人員難以閱讀。

## 3. 文件分層原則

本專案目前採 S / A / B 三層對照：

### S. Sample Version

定位：

- 台灣上市櫃公司常見傳統 SOP / 內控制度寫法示範。
- 作為市場現況 baseline。
- 不作為簡化版、教學版或錯誤版本。
- 不展開 Computer Audit 深度細節。

### A. 治理主文件

定位：

- Level 2 + Level 3 混合型治理文件。
- 以內控、SOP、流程、權責、KCP、表單與 Workflow 為主。
- 給董事會、內稽、財會、營運與工程主管閱讀。
- 不過度展開 Computer Audit 技術細節。

### B. Computer Audit 附加版本

定位：

- 以主文件為基礎。
- 加深 ERP、Workflow、Audit Trail、Log、SoD、異常時序與例外清單。
- 給稽核主管、電腦稽核、資訊與 ERP 相關人員閱讀。
- 不是另一套制度，而是主文件的電腦稽核深度版。

## 4. 導入順序

目前最高原則是循序漸進、治理優先：

```text
內控 / SOP
  -> Computer Audit
  -> Workflow / ERP
  -> 系統留痕與例外清單
  -> 管理追蹤與稽核覆核
```

現階段最多推進到 Computer Audit 與系統留痕檢查，不延伸成更高階自動化。

## 5. 文件輸出規則

核心治理文件原則上同時具備：

- Markdown（`.md`）：便於版本控制、diff、欄位整理、後續系統化。
- Single-file Responsive HTML（`.html`）：便於人類閱讀、教學、展示與內部討論。

Markdown 版本需：

- 結構清楚。
- 標題層級穩定。
- 表格可維護。
- 控制點、表單、欄位、責任與證據鏈可追溯。

HTML 版本需：

- 人類友善。
- RWD。
- 適合教學展示。
- 避免一開始塞滿檔名、代碼與技術欄位。
- 優先用主管看得懂的語氣說明流程、風險、責任與證據。

## 6. 語氣與內容原則

文件應偏：

- 董事會治理。
- 內部稽核。
- 內控制度。
- ERP / Workflow。
- Computer Audit。
- 台灣上市櫃企業可落地的語氣。

避免：

- 空泛口號。
- 過度技術化。
- 對外服務包裝。
- 純系統規格書語氣。
- 在證據不足時使用確定性法律判斷。

## 7. Special Case 原則

「疑似違建 / 未合法申報施工」案例用來測試專案稽核能否發現：

- 文件不一致。
- 時序異常。
- ERP / Workflow 落差。
- 現場與系統不一致。
- Audit Trail 不完整。
- 權責與 SoD 缺口。

本案例不做：

- 自動判定違建。
- 自動執法。
- 自動核准或否決工程款。
- 取代法務、建築師、專業技師、稽核主管或主管機關判斷。

## 9. 最小工作原則

每次修改都應確認：

- 是否更符合公司內部稽核主管使用？
- 是否讓流程、權責、控制點與證據更清楚？
- 是否避免把例外提示寫成最終結論？
- 是否保留人工覆核與專業判斷？
- 是否維持 Markdown / HTML 雙格式可維護？
