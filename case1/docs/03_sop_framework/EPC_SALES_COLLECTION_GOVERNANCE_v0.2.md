# 工程承攬業銷售與收款循環治理作業辦法 v0.2

- 文件狀態：個人工作稿 / v0.2 Draft / 稽核主管校閱版
- 文件定位：Level 2 + Level 3 混合型治理文件
- 文件層級：A 版，傳統治理主文件
- 主要讀者：公司稽核主管、內部稽核、財會、營運、工程主管、業務主管、流程 Owner
- 適用場景：工程承攬業目標單位之銷售與收款循環治理整理
- 對應附加文件：Computer Audit 附加版本

## 0. 文件聲明

本文件不是全集團正式制度，不是最終版，也不是對外服務提案或資訊系統建置規格。

本文件是供公司稽核主管個人使用與內部討論的治理型工作稿。其目的不是追求最完整 SOP，而是建立一份企業可接受、內稽可討論、財會與營運可落地的治理底稿。

本版文件以傳統治理文件為主，重點放在：

- 內部控制。
- SOP。
- 流程。
- 權責。
- 關鍵控制點（KCP）。
- 表單與紀錄。
- Workflow 基礎。
- 文件留存與異常管理。

Computer Audit、ERP Log、Audit Trail、SoD 技術矩陣與 Suspicious Timeline 深度內容，另整理於 Computer Audit 附加版本。這樣可以避免主文件過度技術化，也讓治理導入維持循序漸進。

## 1. 文件分層策略

本主題採一主一附的文件分層。

| 版本 | 文件 | 定位 | 主要讀者 |
| --- | --- | --- | --- |
| A 版 | 傳統治理主文件 | Level 2 + Level 3 混合型治理文件 | 董事會、內稽、財會、營運、工程主管 |
| B 版 | Computer Audit 附加版本 | 電腦稽核深度版 | 電腦稽核、資訊、ERP / Workflow 相關人員 |

兩份文件不是兩套制度。

本文件負責企業可接受與可落地。CA 附加版負責把同一治理主題延伸到電腦稽核、資料追溯與系統留痕檢查。

## 2. 漸進導入原則

治理與電腦稽核導入必須循序漸進。

```text
內控 / SOP
  -> Computer Audit
  -> Workflow / ERP
  -> 系統留痕與例外清單
  -> 管理追蹤與稽核覆核
```

目前階段停在：

> 傳統治理主文件 + Computer Audit 附加版本。

本階段不直接跳到：

- 全自動稽核。
- 系統自動核准。
- 系統自動判斷。
- 對外展示包裝。

## 3. 研究底稿與收斂原則

本文件參考多份研究底稿與工程承攬業治理觀點後整理而成，不以單一來源作為定稿依據。

本版不在正文強調研究工具來源，避免文件讀起來像外部方法論展示。重點是讓稽核主管能直接拿來檢視流程、權責、控制點與證據鏈。

收斂原則：

- 主文件以傳統治理語言呈現。
- Computer Audit 深度內容不塞入主文件正文。
- 不把文件包裝成對外服務或技術展示。
- 保持台灣上市櫃企業 reality，避免過度 ISO 教條化或過度技術化。

## 5. 治理目標

本文件目標是建立工程承攬業銷售與收款循環的可治理基準。

核心目標：

- 確保合約、變更、估驗、請款、收款、驗收與保留款有一致流程。
- 確保權責清楚，避免同一人執行不相容工作。
- 確保關鍵控制點可被管理階層與內部稽核理解與抽核。
- 確保表單、附件與簽核紀錄完整留存。
- 確保 Workflow 可支撐基本授權、覆核與例外呈報。
- 為後續 Computer Audit 與 ERP / Workflow 深化留下乾淨接口。

本文件不追求：

- 全自動稽核。
- 系統自動決策。
- 系統自動核准。
- 一次寫完所有細部作業。
- 以技術名詞取代治理責任。

## 8. 整體流程生命週期

工程承攬業銷售與收款循環不是單純「接單、開票、收款」。

本 Pilot Case 採以下生命週期：

```text
案件機會 / 得標
  -> 合約審查與案件建檔
  -> 契約變更與補充約定
  -> 工程進度與估驗基礎
  -> 工程估驗計價與營收認列
  -> 請款與發票管理
  -> 應收帳款與保留款管理
  -> 收款、沖帳與差異處理
  -> 驗收、結案與尾款釋放
  -> 異常、逾期與爭議處理
  -> 文件留存、監督與改善
```

主文件關注的是流程、權責、控制點與文件證據。跨系統勾稽與資料測試另由 CA 附加版處理。

## 9. 角色與職責

### 9.1 基本 RACI

| 角色 | 主要責任 | 不應兼任 |
| --- | --- | --- |
| 業務 / 專案開發 | 合約條件、客戶溝通、案件啟動 | 收款沖帳、主檔權限維護 |
| 工程 / PM | 進度確認、估驗基礎、驗收協調 | 發票開立、收款沖帳 |
| 財會 | 請款、發票、AR、收款、保留款、財報認列 | 合約條件單獨核准、工程進度單獨認定 |
| 法務 | 合約條款、變更、爭議與重大風險條款 | ERP 主檔維護、收款沖帳 |
| 資訊 / ERP Owner | 系統權限、Workflow 支援、報表支援 | 業務核准、估驗核准 |
| 內部稽核 | 控制設計評估、抽核、缺失追蹤 | 流程執行、交易核准 |
| 管理階層 | 授權、例外核准、監督改善 | 取代必要附件與證據鏈 |

### 9.2 權責原則

- 流程執行人、覆核人與核准人應適度分離。
- 估驗基礎不得只由單一角色自行認定。
- 請款、收款與沖帳應分離。
- 重大合約、重大變更、重大逾期與重大爭議應升級呈報。
- 例外核准不得取代必要佐證文件。

## 10. SC-01 合約審查與案件建檔

### 10.1 程序目的

確保工程承攬案件在啟動前，合約條件、付款里程碑、保留款、違約責任、變更條款、稅務條件與業主風險已完成跨部門審查。

### 10.2 治理要求

- 案件啟動前應完成合約審查。
- 合約金額、付款條件、保留款比例、驗收條件、保固條件應明確。
- 合約附件、圖說、BOQ、報價基礎應納入受控文件。
- 合約核決權限應依金額、風險與條款重大性分層。
- 未完成合約核准前，專案不得進入正式執行或請款流程。

### 10.3 Workflow 基礎

- 合約審查應至少經業務、工程、財會、法務及必要主管簽核。
- 合約主檔建立應以已核准合約為依據。
- 合約重大條款變更應重新送審。

### 10.4 KCP

| KCP | 控制點 | 控制目的 |
| --- | --- | --- |
| KCP-SC-01 | 合約金額與核決層級相符 | 避免未授權合約成立 |
| KCP-SC-02 | 合約主檔與簽核本一致 | 避免系統或台帳與正式合約不一致 |
| KCP-SC-03 | 合約付款條件完整 | 確保後續估驗、請款、保留款有依據 |

### 10.5 表單與紀錄

- FM-SC-01 合約審查表。
- FM-SC-02 案件啟動申請。
- FM-SC-03 合約主檔建立 / 異動申請。
- 合約正本與附件。
- Workflow 簽核紀錄。

## 11. SC-02 工程估驗計價與營收認列

### 11.1 程序目的

確保工程估驗具備實際進度證據、適當覆核與合約基礎，並使估驗、請款、營收認列與應收帳款可追溯。

### 11.2 治理要求

- 估驗需有工程進度依據、業主或監造確認、內部 PM / 工程主管覆核。
- 估驗不得超過合約金額加已核准變更。
- 完工百分比、估驗金額與營收認列基礎需一致。
- 估驗文件需能支撐後續請款、發票與 AR。
- 估驗例外應有原因、責任單位與核准紀錄。

### 11.3 Workflow 基礎

- 估驗申請應由 PM 或工程權責單位提出。
- 財會應檢查估驗是否符合合約付款條件。
- 估驗核准完成前，不得進入正式請款。

### 11.4 KCP

| KCP | 控制點 | 控制目的 |
| --- | --- | --- |
| KCP-SC-04 | 估驗數量經適當確認 | 避免單一人員認定進度 |
| KCP-SC-05 | 累計估驗不超過合約 + 已核准變更 | 避免超額估驗 |
| KCP-SC-06 | 估驗證據完整 | 確保估驗可被內稽與外部查核 |

### 11.5 表單與紀錄

- FM-SC-04 工程估驗申請單。
- FM-SC-05 工程進度確認紀錄。
- FM-SC-06 業主 / 監造確認書。
- 工程照片與施工日報。
- 營收認列底稿。

## 12. SC-03 契約變更與追加減帳

### 12.1 程序目的

確保契約變更、追加減、暫估、工期調整與價金調整均經適當授權，且能回溯調整合約總價與估驗基礎。

### 12.2 治理要求

- 原則上變更應先核准後施工。
- 若因特殊情況需先行施作，需建立受控暫估與例外核准機制。
- 口頭變更不得直接進入正式估驗或請款。
- 變更應有版本、原因、金額、工期與核准層級。
- 變更影響合約價金、估驗、請款或工期時，應通知相關權責單位。

### 12.3 Workflow 基礎

- 變更申請應由工程 / PM 提出，並附原因、金額、工期及佐證。
- 財會應確認變更對估驗、收入與應收帳款之影響。
- 法務應審查重大條款與爭議風險。
- 核准後才可更新合約有效總價。

### 12.4 KCP

| KCP | 控制點 | 控制目的 |
| --- | --- | --- |
| KCP-SC-07 | 變更先核准後施工 | 避免未授權施工進入收入循環 |
| KCP-SC-08 | 變更回溯調整合約總價與估驗基礎 | 確保後續估驗不失準 |
| KCP-SC-09 | 無口頭變更入帳 | 避免文件與實務兩本帳 |

### 12.5 表單與紀錄

- FM-SC-07 工程變更申請單。
- FM-SC-08 追加減明細表。
- FM-SC-09 變更核准紀錄。
- 補充協議或業主確認文件。

## 13. SC-04 請款與發票管理

### 13.1 程序目的

確保請款基於已核准估驗與合約付款條件，並使請款單、發票、AR 與估驗證據一致。

### 13.2 治理要求

- 請款不得早於估驗核准。
- 請款金額不得超過已確認估驗可請款金額。
- 發票應與請款單金額、稅額、專案代號一致。
- 重複請款、孤立請款、未對應估驗之請款應列為例外。
- 發票作廢、折讓或重開應有原因與核准紀錄。

### 13.3 Workflow 基礎

- 請款申請應連結估驗文件與合約付款條件。
- 財會應覆核請款金額、稅額及發票資訊。
- 請款核准後，才可產生正式應收帳款紀錄。

### 13.4 KCP

| KCP | 控制點 | 控制目的 |
| --- | --- | --- |
| KCP-SC-10 | 請款必須對應核准估驗 | 避免無基礎請款 |
| KCP-SC-11 | 請款金額授權核准 | 避免超額或未授權請款 |
| KCP-SC-12 | 請款文件三合一完整 | 確保請款單、估驗單、業主確認可勾稽 |

### 13.5 表單與紀錄

- FM-SC-10 請款申請單。
- FM-SC-11 請款檢核表。
- FM-SC-12 發票開立申請。
- 發票清單。

## 14. SC-05 收款、沖帳與差異處理

### 14.1 程序目的

確保收款及時入帳、正確沖帳，並對差異、扣款、折讓、票據、爭議款與未沖帳款項建立追蹤。

### 14.2 治理要求

- 收款應由財會依銀行入帳或票據資料確認。
- 請款人不得自行執行收款沖帳。
- 差異、扣款、折讓、短收應有核准與原因碼。
- 逾期帳款應定期追蹤。
- 重大逾期、重大差異或爭議款應升級呈報。

### 14.3 Workflow 基礎

- 收款確認與沖帳應留存銀行入帳或票據資料。
- 折讓、扣款與短收應經適當層級核准。
- 未配對收款應進入例外處理流程。

### 14.4 KCP

| KCP | 控制點 | 控制目的 |
| --- | --- | --- |
| KCP-SC-13 | 收款與應收帳款核銷一致 | 避免錯沖或挪用 |
| KCP-SC-14 | 請款與收款職責分離 | 防止自行消滅應收 |
| KCP-SC-15 | 逾期帳款管理 | 確保催收與備抵評估 |

### 14.5 表單與紀錄

- FM-SC-13 收款確認單。
- FM-SC-14 收款沖帳單。
- FM-SC-15 差異 / 折讓 / 扣款核准單。
- 銀行入帳紀錄。
- AR Aging Report。

## 15. SC-06 應收帳款與保留款管理

### 15.1 程序目的

確保正常應收、保留款、尾款、暫估款、爭議款分帳管理，避免狀態混淆造成追蹤失效。

### 15.2 治理要求

- 保留款比例與釋放條件應依合約設定。
- 保留款應建立專案別台帳。
- 到期未請領或未釋放保留款應定期報表追蹤。
- 保留款釋放應與驗收、保固或合約條件連動。
- 爭議款應與正常應收分開管理。

### 15.3 Workflow 基礎

- 保留款釋放需有驗收或合約條件成立證據。
- 到期未請領、未釋放或爭議款應列入追蹤清單。
- 重大保留款釋放應依核決權限簽核。

### 15.4 KCP

| KCP | 控制點 | 控制目的 |
| --- | --- | --- |
| KCP-SC-16 | 保留款比例符合合約 | 避免扣留不足或錯扣 |
| KCP-SC-17 | 保留款台帳完整 | 避免到期漏請或漏釋放 |
| KCP-SC-18 | 保留款釋放條件成立 | 避免未驗收或文件不足即釋放 |

### 15.5 表單與紀錄

- FM-SC-16 保留款台帳。
- FM-SC-17 保留款釋放申請。
- FM-SC-18 尾款追蹤表。
- 驗收或保固相關文件。

## 16. SC-07 驗收、結案與保固管理

### 16.1 程序目的

確保部分驗收、正式驗收、結案、保固啟動、尾款與保留款釋放具備完整文件與狀態連動。

### 16.2 治理要求

- 驗收需有業主、監造、內部工程或相關權責人員確認。
- 缺失改善應列管至結案。
- 未完成驗收不得釋放特定尾款或保留款，除非有例外核准。
- 保固期應依合約啟動並追蹤。

### 16.3 Workflow 基礎

- 驗收申請應檢附驗收文件、缺失改善紀錄與業主確認。
- 驗收狀態應通知財會，以便處理尾款與保留款。
- 保固期、到期日與責任單位應建立追蹤。

### 16.4 KCP

| KCP | 控制點 | 控制目的 |
| --- | --- | --- |
| KCP-SC-19 | 驗收文件完整 | 確保結案依據 |
| KCP-SC-20 | 驗收與保留款釋放連動 | 避免未驗收即釋放 |
| KCP-SC-21 | 保固期建立 | 確保後續維修責任可追蹤 |

### 16.5 表單與紀錄

- FM-SC-19 驗收申請單。
- FM-SC-20 缺失改善追蹤表。
- FM-SC-21 保固書 / 保固台帳。
- 結案證明。

## 17. SC-08 異常、逾期與爭議處理

### 17.1 程序目的

確保客訴、施工爭議、請款退件、逾期款、扣款、品質爭議、法務升級與系統異常有統一異常碼、責任單位與追蹤機制。

### 17.2 治理要求

- 異常事件應建立責任單位、原因、金額、影響、預計處理日期。
- 重大異常應依門檻升級至管理階層。
- 爭議款應與正常應收分開管理。
- 逾期未結案異常應定期呈報。
- 異常結案需有佐證文件。

### 17.3 Workflow 基礎

- 異常事件應有唯一案件編號。
- 異常案件應指定 Owner 與到期日。
- 重大爭議應納入法務或管理階層審查。

### 17.4 KCP

| KCP | 控制點 | 控制目的 |
| --- | --- | --- |
| KCP-SC-22 | 重大異常呈報門檻 | 避免重大風險未升級 |
| KCP-SC-23 | 爭議款凍結或標示 | 避免誤認正常應收 |
| KCP-SC-24 | 異常結案需證據 | 避免口頭結案 |

### 17.5 表單與紀錄

- FM-SC-22 異常事件紀錄單。
- FM-SC-23 異常呈報表。
- FM-SC-24 爭議處理紀錄。
- FM-SC-25 法務意見或會議紀錄。

## 19. 管理監督與內稽抽核

### 19.1 管理監督

管理階層應定期檢視：

- 重大合約與重大變更。
- 累計估驗與請款狀態。
- 逾期應收帳款。
- 保留款到期與釋放狀態。
- 長期未結案異常與爭議。
- 高金額例外核准案件。

### 19.2 內稽抽核

內部稽核可依風險選樣，檢查：

- 合約核准是否完整。
- 估驗是否有進度與業主 / 監造確認。
- 請款是否對應核准估驗。
- 收款沖帳是否正確。
- 保留款釋放是否符合條件。
- 重大異常是否依門檻呈報。

電腦稽核測試、系統資料勾稽、Log 與異常時序分析，請參照 CA 附加版。

## 20. 常見 Red Flags（主文件版）

本章僅列管理層與內稽可快速理解的營運 Red Flags。技術型 Red Flags 另由 CA 附加版展開。

| Red Flag | 風險意義 | 建議動作 |
| --- | --- | --- |
| 未核准合約即啟動專案 | 授權與執行斷裂 | 補查合約審查與案件建檔 |
| 估驗缺少進度或業主 / 監造確認 | 估驗基礎不足 | 要求補證據並追查責任 |
| 變更尚未核准即進入估驗或請款 | 先做後補風險 | 啟動例外審查 |
| 請款金額與估驗基礎不一致 | 可能超額或錯誤請款 | 財會覆核並暫停請款 |
| 收款長期未沖帳 | AR 管理異常 | 追蹤銀行入帳與沖帳責任 |
| 保留款未達條件即釋放 | 合約權益受損 | 追查驗收與保固條件 |
| 異常長期未結案 | 管理監督失效 | 升級呈報 |
