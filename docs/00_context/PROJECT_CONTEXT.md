# Project Context

本文件整理 TAAT x FALO Governance Runtime 的專案背景、角色、目標、限制與導入順序。它是第一層治理骨架，不展開完整 SOP。

## 1. 專案背景

本專案目標是建立 TAAT / FALO 的 AI-native Governance Methodology。

它不是單純做 AI 工具展示，也不是一開始就建置全自動 Agent，而是先從台灣企業可落地的治理需求出發，整理：

- SOP Governance
- Workflow
- Computer Audit
- ERP Mapping
- Audit Trail
- Documentation Runtime
- AI-assisted Governance
- Multi-AI Cross-validation
- HITL

核心觀念：

> 文件不是靜態文件，而是治理 Runtime 的一部分。

文件需要同時承載流程、權責、表單、紀錄、稽核點、系統對應與後續維護邏輯。

## 2. 專案角色

### ff = Force

ff 是人類主導者與最終 HITL。

主要負責：

- 方法論與世界觀
- 治理方向
- 專案優先順序
- TAAT / FALO 架構
- 重大決策與最終確認

### smf = ChatGPT / AI1

smf 是主控 AI，偏向 PL / Architect / Governance Coordinator。

主要負責：

- 任務拆解
- Prompt 與研究設計
- DeepResearch 整合
- Cross-AI Validation 設計
- 方法論收斂
- Workflow 設計

### sxf = Codex / AI2

sxf 是執行 AI，偏向 Execution Agent / Governance Worker / Runtime Builder。

主要負責：

- ERP Mapping 初步整理

## 3. 專案目標

第一階段目標：

- 建立治理文件骨架。
- 建立 C00~C08 方法論索引。
- 建立 Pilot Case 的文件放置位置。
- 明確定義目前階段與不做事項。
- 讓後續 SOP、Computer Audit、Workflow / ERP Mapping 可以有一致入口。

中期目標：

- 完成工程承攬業銷售與收款循環的治理型 SOP 架構。
- 對應 Computer Audit 控制點。
- 對應 ERP 模組、表單、欄位與資料流。
- 建立可追蹤、可維護、可教學的文件體系。

長期方向：

- Digital Auditability
- Evidence Chain
- Workflow Lineage
- AI-assisted Governance
- Cross-AI Validation Runtime
- 未來逐步延伸 AI Audit

## 4. Pilot Case

目前 Pilot Case：

> 工程承攬業銷售與收款循環

本案例暫定涵蓋：

- 客戶資料與信用條件
- 報價與合約
- 工程承攬與進度
- 驗收與請款
- 發票與收款
- 應收帳款管理
- ERP 資料流
- 權責分工
- 稽核軌跡

注意：

此處只定義範圍，不展開完整 SOP。

## 5. 目前階段

目前階段是：

> SOP / Computer Audit / Workflow / ERP Mapping

重點不是先做 AI Automation，而是先建立治理文件與方法論結構。

## 6. 導入順序

建議導入順序：

1. 內控 / SOP
2. Computer Audit
3. Workflow / ERP Mapping
4. Digital Auditability
5. AI-assisted Governance
6. 未來逐步延伸 AI Audit

這個順序代表：

- 先整理人與流程。
- 再整理系統紀錄與稽核點。
- 再整理 ERP 與資料流。
- 最後才逐步加入 AI 輔助與自動化。

## 7. 限制與不做事項

目前不做：

- 不展開完整 SOP 內容。
- 不建立全自動 Agent Runtime。
- 不把 AI 當成最終決策者。
- 不過早產品化 TAAT / FALO。
- 不把 ISO 文件體系硬套到台灣企業內控制度。
- 不把 Multi-AI 當成炫技堆疊。

目前採用：

- Level 2 + Level 3 混合型治理文件。
- 治理優先、漸進導入。
- HITL-first。
- Traceability-first。
- Workflow-first。
