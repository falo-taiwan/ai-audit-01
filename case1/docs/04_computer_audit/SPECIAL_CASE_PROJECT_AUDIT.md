# Special Case Project Audit: 疑似違建 / 未合法申報施工

文件狀態：初版測試案例  
適用範圍：工程承攬專案稽核測試案例  
文件定位：Computer Audit 專案稽核測試案例  
重要聲明：本文件不是法律判決系統，不用來自動認定違建，也不取代法務、建築師、專業技師、內部稽核主管或主管機關之判斷。

## 1. 核心定位

本案例的目的，不是判定某工程是否違建，而是測試工程承攬專案稽核能否發現治理異常。

本案例關注：

- 系統流程看起來正常，但現場 reality 可能有問題。
- ERP / Workflow 已完成簽核，但合法性文件或核准圖說不足。
- 已估驗、已請款、已收款，但施工範圍與核准內容不一致。
- 先施工、後補文件，導致 Audit Trail 無法支撐治理判斷。
- SOP、Computer Audit、Workflow、ERP Mapping 是否能共同發現異常。

本案例不做：

- 系統自動判定違建。
- 系統自動執法。
- 系統自動核准或否決工程款。
- 以單一文件、單一 ERP 欄位或單一照片做最終判斷。

## 2. 專案背景

工程承攬專案通常跨越長週期、多部門、多文件與多系統：

- 合約與報價
- 建照、變更設計、核准圖說或其他合法性文件
- 施工進度與現場紀錄
- 估驗與請款
- 發票、應收帳款與收款
- 驗收、保留款與結案
- ERP / Workflow / DMS / 現場照片 / 會議紀錄

在一般 ERP 或 Workflow 中，專案可能呈現「流程正常」：

1. 專案已建立。
2. 合約已上傳。
3. Workflow 已簽核。
4. 估驗單已核准。
5. 請款單已開立。
6. 發票已開立。
7. 客戶已付款。

但專案稽核需要追問：

> 這些系統紀錄是否真的對應到合法、核准、可追溯的現場施工內容？

## 3. 稽核情境

### 3.1 Special Case 名稱

疑似違建 / 未合法申報施工

### 3.2 情境描述

某工程承攬專案已在 ERP 中建立專案代號，並依 Workflow 完成合約、估驗、請款及收款流程。從系統表面看，流程完整、簽核齊備、帳務也可追溯。

但稽核人員在專案抽核時發現：

- 現場施工照片與核准圖說似乎不一致。
- 估驗項目包含合約外工程或未核准變更。
- 施工日期早於變更單核准日期。
- 請款日期早於合法性文件或核准圖說確認日期。
- ERP 專案範圍與建照 / 變更資料 / 合約附件不一致。
- Workflow 顯示已簽核，但簽核附件不足或版本不明。

### 3.3 典型樣態

- 未申請建照即開工。
- 未申請變更即施工。
- 超出原核准範圍施工。
- 合約外施工，但未建立正式變更單。
- 先施工後補合約或補簽核。
- 現場施工與核准圖說不一致。
- 未正式核准即估驗、請款或收款。
- 使用口頭指示、LINE 訊息或會議結論取代正式 Workflow。

## 4. 常見風險

| 風險類型 | 說明 | 可能造成的治理問題 |
| --- | --- | --- |
| 合法性文件不足 | 建照、變更設計、核准圖說、使用執照或其他必要文件不足 | 工程合法性無法被稽核證明 |
| 合約與施工不一致 | 合約範圍與現場施工項目不一致 | 合約外施工、未授權成本或收入認列風險 |
| 圖說與現場不一致 | 現場照片、施工日報、估驗項目與核准圖說不一致 | 可能存在超範圍施工或未核准變更 |
| 時序異常 | 施工、估驗、請款早於核准或變更日期 | 先做後補、補簽核、Audit Trail 失真 |
| Workflow 形式完整但附件不足 | 簽核完成，但缺少核准圖說、變更單或佐證 | 系統看似合規，實質證據不足 |
| ERP 與現場 reality 落差 | ERP 有帳、有單、有流程，但現場內容不同 | Computer Audit 若只看系統，可能漏掉現場風險 |
| 權責分工失效 | 同一人可建立、核准、估驗、請款或調整主檔 | SoD 失效，增加舞弊或錯誤風險 |

## 5. 專案稽核切入點

本案例應以專案全生命週期作為稽核單位，而不是單一表單或單一 ERP 模組。

### 5.1 專案主檔

稽核問題：

- 專案代號是否唯一？
- 專案主檔是否連到合約、建照、圖說、變更單、估驗單、請款單與驗收文件？
- 專案範圍是否可從 ERP 回推至核准文件？

### 5.2 合約與變更

稽核問題：

- 合約範圍是否清楚定義施工項目、位置、面積、樓層、數量或圖說版本？
- 合約外施工是否有正式變更單？
- 變更單日期是否早於施工、估驗與請款？

### 5.3 合法性文件與核准圖說

稽核問題：

- 建照、雜項執照、變更設計、核准圖說或其他必要文件是否存在？
- 文件版本是否與施工期間相符？
- ERP / Workflow 是否有欄位記錄合法性文件編號與版本？

### 5.4 現場施工與估驗

稽核問題：

- 估驗項目是否對應合約與核准圖說？
- 估驗照片、施工日報、監造紀錄是否支撐估驗內容？
- 是否有未核准變更卻已估驗的情形？

### 5.5 請款、收款與驗收

稽核問題：

- 請款是否基於已核准估驗？
- 驗收是否基於合法、核准、可追溯的施工範圍？
- 保留款釋放是否有驗收或合法性文件支撐？

## 6. Audit Questions

### 6.1 合法性文件

- 本專案是否需要建照、雜項執照、變更設計或其他主管機關文件？
- 文件是否在施工前或施工中依規定取得？
- 文件編號、核准日期、圖說版本是否可在 ERP / Workflow / DMS 中查到？
- 若文件不存在，是否有明確的免辦依據或法務 / 專業人員判斷紀錄？

### 6.2 合約與施工一致性

- 合約約定範圍是否等於現場施工範圍？
- 合約附件圖說是否與核准圖說一致？
- 合約外工程是否有正式變更單與授權？
- 是否有先施工後簽約或先施工後變更的紀錄？

### 6.3 圖說與現場一致性

- 現場照片是否能對應圖說版本？
- 施工日報是否標示施工位置、數量、日期與責任人？
- 估驗項目是否能回溯到圖說、施工日報與現場照片？

### 6.4 ERP / Workflow 一致性

- ERP 專案主檔是否記錄建照或核准文件欄位？
- Workflow 是否要求上傳合法性文件或核准圖說？
- 是否允許未上傳文件仍進入估驗或請款？
- 是否存在管理者後台修改欄位但未留下完整 Log？

### 6.5 Audit Trail 完整性

- 誰建立專案？
- 誰上傳合約與合法性文件？
- 誰確認圖說版本？
- 誰核准估驗？
- 誰核准請款？
- 誰釋放保留款？
- 每個節點是否有時間戳、版本與附件？

## 7. KCP: Key Control Points

| KCP | 控制點 | 控制目的 | Computer Audit 測試方向 |
| --- | --- | --- | --- |
| KCP-SC-01 | 專案合法性文件建檔 | 確保施工前具備必要申報或核准文件 | 檢查專案主檔是否有文件編號、核准日期、版本與附件 |
| KCP-SC-02 | 合約範圍與核准圖說一致 | 避免合約、現場、圖說三者脫節 | 比對合約附件圖說版本與核准圖說版本 |
| KCP-SC-03 | 變更先核准後施工 | 防止先施工後補文件 | 比對變更單核准日期、施工日報日期、估驗日期 |
| KCP-SC-04 | 估驗必須對應核准範圍 | 防止未核准施工被估驗 | 比對估驗項目、合約項目、圖說區域與現場照片 |
| KCP-SC-05 | 請款必須對應核准估驗 | 防止未核准工程進入收款循環 | 檢查請款單是否有對應估驗單與附件 |
| KCP-SC-06 | Workflow 附件完整性 | 避免形式簽核但實質證據不足 | 檢查必要附件是否缺漏、補傳或版本不一致 |
| KCP-SC-07 | SoD 權限分離 | 防止同一人建立、核准、請款、調整資料 | 比對角色權限與實際操作 Log |
| KCP-SC-08 | 保留款釋放條件 | 確保驗收與合法性文件完整後才釋放 | 比對驗收日期、使用/驗收文件、保留款釋放日期 |

## 8. Red Flags

### 8.1 時序 Red Flags

- 施工日期早於建照或合法性文件日期。
- 施工日期早於變更單核准日期。
- 估驗日期早於變更單核准日期。
- 請款日期早於估驗核准日期。
- 驗收日期早於圖說或變更文件補正日期。
- 保留款釋放日期早於驗收或結案文件日期。

### 8.2 文件 Red Flags

- 合法性文件欄位空白，但專案已進入估驗或請款。
- 圖說版本不明，或同一專案存在多個未標示版次的圖說。
- Workflow 附件在簽核後補傳。
- 合約附件與估驗附件名稱不同但無版本對照。
- 現場照片無日期、無地點、無對應施工項目。

### 8.3 ERP / Workflow Red Flags

- 專案主檔可建立但未強制填入合法性文件欄位。
- 變更單未核准仍可估驗。
- 管理者或 IT 帳號直接修改合約金額、估驗狀態或文件欄位。
- 同一人具有專案建檔、估驗核准、請款核准與主檔維護權限。
- Workflow 顯示完成，但附件清單不完整。

### 8.4 現場 Reality Red Flags

- 現場照片顯示的施工範圍超出合約或圖說。
- 施工日報描述與 ERP 估驗項目不同。
- 估驗數量與現場照片、監造紀錄不一致。
- 現場已施工，但系統中的變更單仍為草稿或退回狀態。

## 9. Suspicious Timeline: 可疑時序

Suspicious Timeline 是本案例的核心概念。

目的：

> 將施工、變更、簽核、估驗、請款、驗收等日期放在同一條時間軸上，檢查是否存在「先做、後補、時序異常」。

### 9.1 最小時間軸欄位

| 時間欄位 | 來源 | 稽核意義 |
| --- | --- | --- |
| 合約簽核日期 | Workflow / DMS | 確認專案授權起點 |
| 建照或合法性文件日期 | DMS / 專案主檔 | 確認施工合法性文件基礎 |
| 核准圖說版本日期 | DMS / 圖說管理 | 確認施工依據版本 |
| 變更單建立日期 | ERP / Workflow | 確認變更是否被正式提出 |
| 變更單核准日期 | Workflow | 確認變更是否先核准 |
| 施工日報日期 | 現場系統 / 附件 | 確認實際施工時間 |
| 估驗申請日期 | ERP | 確認工程被轉入計價的時間 |
| 估驗核准日期 | Workflow | 確認估驗是否經覆核 |
| 請款日期 | ERP / AR | 確認對外請款時間 |
| 收款日期 | ERP / 銀行對帳 | 確認現金流入時間 |
| 驗收日期 | Workflow / 驗收文件 | 確認結案與保留款條件 |

### 9.2 可疑時序規則

| 規則 | 可疑情境 |
| --- | --- |
| 施工日 < 合法性文件核准日 | 可能未合法申報即施工 |
| 施工日 < 變更單核准日 | 可能先施工後補變更 |
| 估驗日 < 變更單核准日 | 未核准變更可能已進入估驗 |
| 請款日 < 估驗核准日 | 請款基礎可能不足 |
| 收款日 < 請款核准日 | 收款與帳務流程可能異常 |
| 保留款釋放日 < 驗收日 | 保留款釋放條件可能未成立 |

## 10. Workflow / ERP 關聯

### 10.1 ERP 模組

- Project Master：專案主檔、合法性文件欄位、專案狀態。
- Contract Module：合約、合約附件、合約版本。
- Change Order Module：變更單、追加減、核准狀態。
- Valuation Module：估驗單、估驗項目、估驗金額。
- AR / Billing Module：請款單、發票、應收帳款。
- Collection Module：收款、沖帳、銀行對帳。
- Retention Module：保留款扣留與釋放。

### 10.2 Workflow 節點

- 合約審查
- 合法性文件確認
- 圖說版本確認
- 變更單核准
- 估驗核准
- 請款核准
- 驗收核准
- 保留款釋放
- 異常事件呈報

### 10.3 建議系統控制

- 未填合法性文件欄位，不得進入施工/估驗節點。
- 未核准變更單，不得估驗變更項目。
- 未上傳圖說版本，不得核准估驗。
- 未完成驗收，不得釋放保留款。
- Workflow 附件補傳應留下補傳時間與補傳人。
- IT 或管理者後台修正應進入特權操作 Log。

## 11. 文件與證據鏈

### 11.1 證據鏈最小集合

| 類型 | 文件 / 資料 | 目的 |
| --- | --- | --- |
| 合法性文件 | 建照、雜項執照、變更設計、使用執照、免辦判斷紀錄 | 判斷施工是否具備必要文件基礎 |
| 合約文件 | 合約、附件、報價單、BOQ、付款條件 | 對應施工範圍與請款條件 |
| 圖說文件 | 核准圖說、施工圖、竣工圖、版本對照表 | 對應現場與估驗內容 |
| 現場證據 | 施工日報、照片、監造紀錄、會議紀錄 | 對應實際施工時間與範圍 |
| ERP 紀錄 | 專案主檔、變更單、估驗單、請款單、收款紀錄 | 對應系統資料鏈 |
| Workflow 紀錄 | 簽核路徑、核准時間、退回紀錄、附件清單 | 對應授權與時序 |
| Audit Trail | 建立、修改、刪除、補件、後台修正 Log | 對應可追溯性與防竄改 |

### 11.2 證據鏈原則

- 每一筆估驗應能追溯到合約、圖說、施工日報與現場證據。
- 每一筆請款應能追溯到核准估驗。
- 每一筆變更施工應能追溯到變更單核准。
- 每一次後補文件都應留下補件時間與補件理由。
- 若無法形成完整證據鏈，應列為稽核發現或待釐清事項。

## 12. 與法規背景的關係

本文件只作為專案稽核測試案例，不作為法律意見。

背景上，《建築法》對建築許可、建造執照、雜項執照、使用執照、圖說與施工一致性、變更設計、使用管理等事項有明確規範。稽核人員在實際個案中，應依專案所在地、建物類型、主管機關要求、合約條件與法務 / 建築專業判斷進一步確認。

## 13. 本案例測試目的

本案例用來測試：

- Computer Audit 是否能發現時序異常。
- ERP / Workflow 是否可追溯現場與文件。
- SOP 是否能要求必要文件與控制點。
- Audit Trail 是否完整保留關鍵證據。
- Computer Audit 是否能協助整理異常與證據鏈。
- 稽核主管與稽核人員是否能基於完整資料做出專業判斷。

結論：

> 本測試案例的價值，不在於自動判定違建，而在於讓專案稽核更早看見治理異常、文件斷點、時序風險與系統落差。
